## 📌 Project Overview

Governments receive thousands of citizen complaints every day — about broken roads, water shortages, power outages, sanitation failures, and public safety concerns. Today, these are handled manually: officials read each complaint, guess which department should handle it, and then file it into a queue. The result is delays, misrouting, and ignored emergencies.

This project builds an **end-to-end NLP pipeline** that:
- **Reads** raw, unstructured citizen complaints in plain English
- **Cleans and normalizes** the text automatically
- **Routes** each complaint to the correct government department (Water, Electricity, Roads, Sanitation, Public Safety)
- **Scores the urgency** so critical issues surface immediately
- **Exposes the models as an API** for integration with government portals

The four-week roadmap progressively builds from raw data → cleaned text → classified complaints → sentiment-scored → deployed API.

## Week 1 — Data Collection, Text Cleaning & EDA

**Notebook:** `Week1_EDA_Grievance_NLP.ipynb`

### Objective

Before any model can be built, the raw text must be understood and cleaned. Week 1 establishes the data foundation: loading the dataset, exploring its structure, preprocessing the complaints, and visualizing patterns that will guide model design in later weeks.

### Dataset

The project uses either the **NYC 311 Service Request dataset** (available on Kaggle) or a **synthetic dataset** generated within the notebook. The synthetic generator covers 5 departments and produces realistic complaint narratives:

| Department | Sample Complaint |
|---|---|
| Water Supply | *"Water supply is completely cut off in our area for 5 days."* |
| Electricity | *"Frequent voltage fluctuations are damaging home appliances."* |
| Roads & Transport | *"Massive potholes on main road have damaged several vehicles."* |
| Sanitation | *"Garbage has not been collected for 7 days. Stench is unbearable."* |
| Public Safety | *"Stray dogs are attacking children near the school."* |

Key columns used:
- `narrative` — raw complaint text written by the citizen
- `product` — the target department label (used for supervised training)

### Pipeline Steps

#### Step 1 — Library Setup
Installs and imports all required packages: `nltk`, `spacy`, `wordcloud`, `matplotlib`, `seaborn`, `pandas`, `scikit-learn`. Downloads the `en_core_web_sm` language model for spaCy.

#### Step 2 — Load Dataset
Reads the dataset from CSV. If `complaints.csv` is not present, the synthetic generator creates a balanced dataset of 1,250 complaints (250 per department).

#### Step 3 — Exploratory Data Analysis (EDA)
Examines the dataset before any transformation:
- `.info()` — column types and null counts
- `.isnull().sum()` — identifies missing values (filled with column mode)
- `word_count` and `text_length` distributions — histograms showing how long complaints are
- Average word count by department — reveals which departments get longer or shorter complaints
- Department distribution bar chart — checks for class imbalance

> **Why this matters:** Class imbalance means a model could achieve high accuracy simply by predicting the majority class every time. Detecting this early guides decisions about sampling strategy and evaluation metrics.

#### Step 4 — Text Preprocessing
Applies a multi-stage cleaning pipeline to every complaint:

| Stage | Operation | Example |
|---|---|---|
| Lowercasing | `"Water SUPPLY"` → `"water supply"` | Standardizes capitalization |
| URL removal | `"see http://gov.in"` → `"see "` | Strips irrelevant links |
| Special character removal | `"road!! is bad..."` → `"road is bad"` | Keeps only alphabetic tokens |
| Whitespace normalization | Multiple spaces → single space | Cleans up formatting artifacts |
| Lemmatization (spaCy) | `"running"` → `"run"`, `"damaged"` → `"damage"` | Reduces words to base forms |
| Stopword removal (NLTK) | Removes `"the"`, `"is"`, `"in"`, `"a"` etc. | Drops non-informative words |

The result is stored in a new column `processed_text` — the form used for all downstream modeling.

**Why lemmatize vs stem?**
Lemmatization (used here via spaCy) produces real dictionary words (`"ran"` → `"run"`), while stemming produces truncated roots (`"ran"` → `"ran"`). Lemmatization is slower but more accurate, which matters for short civic complaint text.

#### Step 5 — Word Cloud Visualization
Generates word clouds for:
- **Overall corpus** — shows the most frequent terms across all complaints
- **Per-department** — shows unique vocabulary for each department category

These visually confirm that the preprocessing is working (civic keywords like `"water"`, `"garbage"`, `"road"` should dominate, not stopwords).

#### Step 6 — N-gram Frequency Analysis
Plots the top 15 most frequent:
- **Unigrams** (single words) — overall most common terms
- **Bigrams** (two-word phrases) — captures `"water supply"`, `"power outage"`, `"garbage collected"`
- **Trigrams** (three-word phrases) — captures `"street light broken"`, `"no water days"`

> Bigrams and trigrams reveal compound civic terms that unigrams miss. `"water"` alone is ambiguous; `"water supply"` is not.

#### Step 7 — Save Processed Data
Saves the cleaned dataframe as `processed_grievances.csv`. This file is the handoff artifact to Week 2.

## Week 2 — Topic Modeling & Department Categorization

**Notebook:** `Week2_Classification_Grievance_NLP.ipynb`

### Objective

With clean text available, Week 2 converts it into numbers (via TF-IDF) and trains multiple classification models to automatically route a raw complaint to its correct government department. The goal is to select the best model using cross-validated Macro F1-Score and save it for Week 3's API layer.

### Why Macro F1 and not Accuracy?

Accuracy is misleading when classes are imbalanced. If 80% of complaints are about Water Supply, a model that always predicts "Water Supply" gets 80% accuracy — but is useless. **Macro F1** averages the F1-score across each class equally, meaning a model must perform well on every department, including minority ones, to score highly.

### Pipeline Steps

#### Step 1 — Load Data
Loads `processed_grievances.csv` from Week 1. If the file is missing, the notebook auto-generates the same synthetic dataset so it can run standalone.

#### Step 2 — Label Encoding
Converts string department names into integer class labels using `LabelEncoder`:
```
0 → Electricity
1 → Public Safety
2 → Roads & Transport
3 → Sanitation
4 → Water Supply
```

#### Step 3 — Train / Test Split
Splits data 80% / 20% using **stratified sampling** — meaning each split contains the same proportion of each department class as the full dataset. This prevents a lucky or unlucky split from distorting results.

#### Step 4 — TF-IDF Vectorization

**What is TF-IDF?**

TF-IDF stands for Term Frequency–Inverse Document Frequency. It assigns a numerical weight to every word in every document:

- **TF (Term Frequency):** How often a word appears in *this* complaint. Words that appear more get higher weight.
- **IDF (Inverse Document Frequency):** How rare a word is across *all* complaints. Words that appear in every complaint (like "the") get heavily penalized.

The combined score rewards words that are frequent in one document but rare across the corpus — exactly the signal needed to distinguish departments.

**Configuration used:**

| Parameter | Value | Reason |
|---|---|---|
| `ngram_range=(1,2)` | Unigrams + bigrams | Captures `"water supply"` as one feature, not two |
| `max_features=10000` | Top 10,000 features | Keeps the vocabulary manageable |
| `sublinear_tf=True` | Log-scale TF | Prevents a word appearing 100x from being 100x more important than one appearing 10x |
| `min_df=2` | Minimum 2 documents | Drops rare typos and one-off words |

The output is a sparse matrix of shape `(n_complaints × 10000)` — each complaint is now a numerical vector.

A bar chart shows the **top 10 TF-IDF keywords per department**, visually confirming the vectorizer has learned discriminative vocabulary.

#### Step 5 — Baseline Models

**Multinomial Naive Bayes**
A probabilistic classifier that assumes each word contributes independently to the probability of a class. Despite this simplistic assumption, it performs surprisingly well on text data and is extremely fast to train. Good baseline to beat.

**Logistic Regression**
Fits a linear decision boundary between classes using log-odds. In text classification, this is a strong baseline — often competitive with much more complex models. Uses `multi_class='multinomial'` for true one-vs-all multi-class handling.

#### Step 6 — Advanced Models

**Linear SVM (LinearSVC)**
Support Vector Machines find the hyperplane that maximally separates classes in high-dimensional feature space. `LinearSVC` is optimized for sparse matrices and typically achieves the best performance on TF-IDF features. It is the standard benchmark for text classification tasks.

**Random Forest**
An ensemble of 200 decision trees. Each tree votes, and the majority wins. More powerful than a single tree, less prone to overfitting. Slower to train but useful for comparison and for later feature importance analysis.

#### Step 7 — Stratified 5-Fold Cross-Validation

Cross-validation guards against the risk that a single train/test split was unusually lucky or unlucky.

**How it works:**
1. The full dataset is split into 5 equal folds
2. For each fold: train on the other 4 folds, evaluate on this fold
3. Repeat 5 times so every sample is in the test set exactly once
4. Report mean ± standard deviation of Macro F1

**Stratified** means each fold preserves the original class proportions — preventing a fold from accidentally having zero samples of one department.

A bar chart with error bars visualizes the mean CV Macro F1 and variance for each model.

#### Step 8 — Hyperparameter Tuning with GridSearchCV

Rather than tuning TF-IDF and the model separately, a **`Pipeline`** chains them together so GridSearchCV can tune both simultaneously:

```
Pipeline: TfidfVectorizer → LinearSVC
```

Grid searched parameters:

| Parameter | Values Tried |
|---|---|
| `tfidf__ngram_range` | `(1,1)` vs `(1,2)` |
| `tfidf__max_features` | `5000` vs `10000` |
| `clf__C` (SVM regularization) | `0.1`, `1.0`, `5.0` |

This tests 2 × 2 × 3 = **12 configurations**, each evaluated with 5-fold CV → 60 total fits. The best configuration is selected automatically and evaluated on the held-out test set.

> **What does `C` control?** The regularization strength. Low `C` creates a wider, more forgiving margin (underfits slightly). High `C` creates a tighter margin that tries harder to classify all training examples correctly (can overfit). The best value depends on the data.

#### Step 9 — Model Evaluation

For the best model, the notebook generates:

**Classification Report** — per-department breakdown of:
- **Precision:** Of all complaints routed to this department, what fraction actually belonged there?
- **Recall:** Of all complaints that belong to this department, what fraction were correctly routed?
- **F1-Score:** Harmonic mean of precision and recall

**Confusion Matrix** — a heatmap grid showing exactly which departments get confused with which. Off-diagonal cells reveal systematic misclassifications (e.g., does the model confuse Sanitation with Public Safety?).

**Model Comparison Bar Chart** — grouped bars showing Accuracy, Macro F1, and Weighted F1 for all models side by side.

Demonstrates the full end-to-end routing flow on 6 unseen complaints:

```

```

Example:

## 🔑 Key Concepts Glossary

| Term | Plain English Explanation |
|---|---|
| **Tokenization** | Splitting `"Water supply broken"` into `["Water", "supply", "broken"]` |
| **Lemmatization** | Reducing `"running"` → `"run"`, `"damaged"` → `"damage"` |
| **Stopwords** | Common words like `"the"`, `"is"`, `"a"` that carry no meaning |
| **TF-IDF** | A number representing how important a word is to one document vs all documents |
| **N-gram** | A sequence of N consecutive words — bigram: `"water supply"`, trigram: `"no water days"` |
| **Macro F1** | Average F1 across all classes, treating each class equally regardless of size |
| **Cross-Validation** | Testing the model on multiple different train/test splits to get reliable accuracy |
| **GridSearchCV** | Automatically trying every combination of hyperparameters and picking the best one |
| **Stratified Split** | Ensuring every train/test split has the same class proportions as the full dataset |
| **Confusion Matrix** | A table showing what the model predicted vs what was actually correct |
| **Pipeline** | Chaining TF-IDF + classifier so they can be trained, tuned, and deployed as one unit |
